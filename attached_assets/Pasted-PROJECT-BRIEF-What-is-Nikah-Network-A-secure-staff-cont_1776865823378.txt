PROJECT BRIEF
What is Nikah Network?
A secure, staff-controlled matrimonial web platform for Falah Khandan Center (Women Commission, Lahore). It digitalizes the manual matchmaking process (paper/WhatsApp) while maintaining Islamic values, privacy, and family involvement.

Core Principle
Staff approves everything. No user can contact another user without staff approval. No profile goes live without staff review.

User Roles
Applicants (Public Users) - Register, view profiles, send proposals, communicate only via approved Q&A
Staff (Admin) - Approve profiles, review matches, approve proposals, moderate Q&A, manage counselling
Management (Optional) - View dashboards, reports, KPIs
6 Core Features (MVP)
Registration & Profile Management - Self-registration + staff profile creation + approval workflow
Duplicate Detection - Email/phone exact match + fuzzy name/DOB matching
AI-Assisted Matching - Hard filters + weighted scoring (transparent breakdown)
Proposal System - Send proposal → staff approval → Q&A communication (moderated, time-bound)
Admin/Staff Dashboard - Profile approvals, match reviews, proposal tracking, audit logs
Counselling Requests - Pre/post-marriage counselling request system
Comparison to Muzz
Feature	Nikah Network	Muzz
Moderation	100% staff-controlled	User-controlled with community moderation
Communication	Q&A only (staff-reviewed)	Open chat (flagged content removed)
Approvals	Every profile, proposal, message	Verified on signup only
Islamic Focus	Counselling built-in, family involvement	Dating-app style
Target	Institutional (centers, families)	Individual users

Flow 2: Matching (Automatic + Staff Review)
Code
1. Automatic Match Generation (Daily Cron, 2 AM)
   - System iterates all "approved" profiles
   - For each profile A:
     * Find all profiles B where:
       - Gender opposite A
       - Age within A's preference range
       - Education ≥ A's minimum
       - Income ≥ A's minimum
       - Location within radius
       - Caste in A's accepted list
       - Marital status accepted
       - Children count acceptable
     * Calculate match score for A→B and B→A using formula:
       score = (ageMatch × 0.20) + (locationMatch × 0.20) + 
               (casteMatch × 0.25) + (educationMatch × 0.15) + 
               (incomeMatch × 0.20)
     * If score ≥ 65: save Match record (suggested)
   
2. Staff Reviews Matches in Dashboard
   - Staff logs in → "Pending Matches" tab
   - Shows: Profile A → Suggested Profile B (score: 78/100)
   - Score breakdown visible:
     * Age: 90/100, Location: 100/100, Caste: 80/100, Education: 65/100, Income: 60/100
   - Staff can:
     * "Approve" (match becomes active)
     * "Hide" (staff decides match is unsuitable)
     * "Reject" (ask system to recalculate this pair)
   - Audit log: "Staff [name] approved match [A] ↔ [B]"
   
3. User Views Matches
   - User logs in → "My Matches" tab
   - Shows top 5 approved matches (staff-approved only)
   - Each match shows:
     * Photo (if approved by staff)
     * Name, Age, City, Education, Occupation
     * Match Score: 78/100 with breakdown (transparent)
     * "Send Proposal" button
   - User can click profile to see full details
   - User can also mark match as "Not Interested" (hidden for user, not for staff)
   
4. System Recalculates Weekly
   - New profiles added → generate matches for them
   - Preferences updated → regenerate matches for that user
   - Manual trigger: Admin can force recalculation
Matching Algorithm Pseudocode:

TypeScript
function calculateScore(profileA: Profile, profileB: Profile): Match {
  const weights = {
    age: 0.20,
    location: 0.20,
    caste: 0.25,
    education: 0.15,
    income: 0.20
  };
  
  // Hard filters (disqualify if not met)
  if (!isAgeInRange(profileB, profileA.preferences.ageRange)) return null;
  if (!isLocationInRadius(profileA, profileB, profileA.preferences.locationRadius)) return null;
  if (!isCasteAccepted(profileB, profileA.preferences.castePreferences)) return null;
  if (profileB.education < profileA.preferences.educationMin) return null;
  if (profileB.monthlyIncome < profileA.preferences.incomeMin) return null;
  if (!isMaritalStatusAccepted(profileB, profileA.preferences.maritalStatusAccepts)) return null;
  if (!areChildrenAccepted(profileB, profileA.preferences.childrenAccepts)) return null;
  
  // Soft scoring (0-100 per category)
  const ageScore = 100 - (Math.abs(profileA.age - profileB.age) / 20 * 100);
  const locationScore = (maxDistance / distance(profileA.city, profileB.city)) * 100;
  const casteScore = profileA.preferences.castePreferences.includes(profileB.caste) ? 100 : 30;
  const educationScore = educationHierarchy[profileB.education] >= educationHierarchy[profileA.preferences.educationMin] ? 100 : 70;
  const incomeScore = (profileB.monthlyIncome / profileA.preferences.incomeMin) * 100;
  
  const totalScore = Math.min(100, Math.round(
    ageScore * weights.age +
    locationScore * weights.location +
    casteScore * weights.caste +
    educationScore * weights.education +
    incomeScore * weights.income
  ));
  
  return {
    score: totalScore,
    breakdown: { ageScore, locationScore, casteScore, educationScore, incomeScore }
  };
}
Flow 3: Proposal & Moderated Q&A
Code
1. User Sends Proposal
   - User clicks "Send Proposal" on a match
   - Optional: User writes intro message (max 200 chars)
   - Click "Send"
   - Status: Proposal created, "pending_staff_approval"
   - Both users notified: "New proposal request (awaiting staff review)"
   
2. Staff Reviews & Approves Proposal
   - Staff dashboard → "Pending Proposals" tab
   - Shows: [User A] wants to contact [User B]
   - Staff verifies both profiles are active + no red flags
   - Staff clicks "Approve" or "Reject"
   - If rejected: add reason (e.g., "user A has negative history")
   - Audit log: "Staff [name] approved proposal [A] → [B]"
   
3. Q&A Communication Opens
   - Both users notified: "Proposal approved! You can now ask questions."
   - Communication window: 14 days (configurable)
   - Counter-proposal not needed; both can ask/answer simultaneously
   
4. User Asks Question
   - User A clicks "Ask Question"
   - Types question (max 300 chars), e.g., "What do you expect from a spouse?"
   - Clicks "Send"
   - Status: Question "pending_staff_review" (not visible to User B yet)
   - Staff inbox updated
   
5. Staff Reviews Question
   - Staff dashboard → "Message Moderation Queue"
   - Shows question from User A
   - Staff checks for:
     * Respectful language
     * Not overly personal (e.g., "what's your salary?" → allowed; "are you a virgin?" → flagged)
     * No contact info (phone numbers, social media)
   - Staff clicks "Approve" or "Reject"
   - If rejected: question never shown to User B, User A notified (reason given)
   - Audit log: "Staff [name] approved message in proposal [A] → [B]"
   
6. User B Sees Question & Replies
   - User B sees approved question from User A
   - User B types reply (max 500 chars)
   - Clicks "Send"
   - Status: Reply "pending_staff_review"
   - Notification sent to User A: "New reply to your question (awaiting staff review)"
   
7. Staff Reviews Reply
   - Same moderation process as questions
   - If approved: appears in User A's Q&A thread
   - If rejected: User B notified with reason
   
8. Q&A Conversation Continues
   - Max 10 questions per user (configurable)
   - Each question can have 1 reply per user
   - Counter-questions allowed
   - Timeline: 14 days from approval (auto-closes)
   - Either user can click "Decline" to close early (irreversible)
   
9. Proposal Outcome
   - After conversation: User A or User B clicks:
     * "Accept" (we want to take this to next level → transfer to family/counselling)
     * "Decline" (not interested)
   - Other user notified
   - Proposal status updated
   - Audit log: "Proposal [A] → [B] closed with outcome: [accepted/declined]"
   
10. Post-Proposal
    - Both users can request counselling (pre-marriage)
    - Staff can remove proposal from system after 30 days (privacy)

SECURITY ARCHITECTURE
Authentication & Authorization
Code
1. JWT-Based Authentication
   - Token payload: { userId, role, email, iat, exp }
   - Access token: 24 hours
   - Refresh token: 7 days (HTTP-only cookie)
   - Middleware: verifyToken() checks JWT, extracts userId/role

2. Password Security
   - Hashed with bcrypt (rounds: 12)
   - Min 8 chars, 1 uppercase, 1 number, 1 special char
   - Never stored in logs

3. 2FA (Staff Only)
   - TOTP via speakeasy (Google Authenticator, Authy)
   - Backup codes generated on setup
   - Required on every staff login

4. Role-Based Access Control (RBAC)
   - Middleware: requireRole("admin" | "staff" | "applicant")
   - Routes protected at controller level
   - Example: PATCH /api/staff/profiles/:id only if role: "staff"
Data Protection
Code
1. Encryption at Rest
   - Sensitive fields encrypted: phone (in communications), totpSecret
   - Database: TLS connection required (MongoDB Atlas)
   - Storage: Cloudinary HTTPS + encryption

2. Encryption in Transit
   - All APIs: HTTPS only
   - HSTS headers enabled
   - Cookies: secure, httpOnly, sameSite=strict

3. Data Masking
   - Applicant email/phone hidden until proposal approved
   - Staff can see unmasked data (audit logged)
   - Messages moderated before display

4. Audit Logging
   - Every staff action logged: { staffId, action, resourceId, timestamp, ipAddress }
   - Immutable log (cannot be edited/deleted)
   - Retention: 2 years (MongoDB TTL index)

5. GDPR Compliance
   - Right to export: /api/users/export (JSON dump of user data)
   - Right to delete: /api/users/delete (soft delete, 30-day grace)
   - Consent tracking: consentGiven, consentGivenAt
   - Data retention policy: auto-delete after 2 years of inactivity
API Security
Code
1. Rate Limiting
   - Auth endpoints: 5 req/min per IP (slow down brute force)
   - General API: 100 req/min per user
   - File upload: 1 file/user/day

2. Input Validation
   - Zod schemas on all endpoints
   - Max string lengths enforced (e.g., bio: 500 chars)
   - Email/phone validated with regex + unique checks

3. SQL/NoSQL Injection Prevention
   - Mongoose (parameterized queries)
   - No dynamic query building
   - Zod whitelist allowed fields on PATCH

4. CORS
   - Frontend domain whitelist: REACT_APP_API_URL
   - Credentials: "include" for cookies
   - Methods: GET, POST, PATCH, DELETE only

5. CSRF Protection
   - SameSite=Strict cookies
   - CSRF tokens optional (stateless JWT sufficient)

6. XSS Prevention
   - React sanitizes by default
   - User-generated text (bio, Q&A) sanitized with DOMPurify
   - No dangerouslySetInnerHTML used

7. Security Headers
   - helmet.js middleware
   - X-Frame-Options: DENY
   - X-Content-Type-Options: nosniff
   - Strict-Transport-Security: max-age=31536000
Sensitive Fields
Code
NEVER expose in API responses:
- passwordHash
- totpSecret
- refreshToken
- emailVerificationToken
- resetPasswordToken

MASKED in communications:
- phone (show last 4 digits only: ****5555)
- email (show part: a***@gmail.com)

INTERNAL ONLY (staff):
- auditLogs
- duplicateDetectionDetails
- rejectionReasons (visible to user if rejected)

// server/src/models/User.ts
import mongoose, { Schema, Document } from 'mongoose';
import bcryptjs from 'bcryptjs';

interface UserProfile {
  firstName: string;
  lastName: string;
  dob: Date;
  gender: 'M' | 'F';
  caste: string;
  education: string;
  occupation: string;
  monthlyIncome: number;
  city: string;
  photoUrl?: string;
  bioData?: string;
  preferences?: {
    ageRange: [number, number];
    locationRadius: number;
    castePreferences: string[];
  };
}

interface IUser extends Document {
  email: string;
  phone: string;
  passwordHash: string;
  role: 'admin' | 'staff' | 'applicant';
  profile?: UserProfile;
  status: 'active' | 'inactive' | 'suspended';
  profileStatus: 'pending' | 'approved' | 'rejected';
  totpSecret?: string;
  totpEnabled: boolean;
  emailVerified: boolean;
  consentGiven: boolean;
  createdAt: Date;
  updatedAt: Date;
  comparePassword(password: string): Promise<boolean>;
}

const userSchema = new Schema<IUser>(
  {
    email: { type: String, unique: true, lowercase: true, required: true, index: true },
    phone: { type: String, unique: true, required: true, index: true },
    passwordHash: { type: String, required: true },
    role: { type: String, enum: ['admin', 'staff', 'applicant'], default: 'applicant' },
    profile: {
      firstName: String,
      lastName: String,
      dob: Date,
      gender: { type: String, enum: ['M', 'F'] },
      caste: String,
      education: String,
      occupation: String,
      monthlyIncome: Number,
      city: String,
      photoUrl: String,
      bioData: String,
      preferences: {
        ageRange: [Number],
        locationRadius: Number,
        castePreferences: [String],
      },
    },
    status: { type: String, enum: ['active', 'inactive', 'suspended'], default: 'active' },
    profileStatus: { type: String, enum: ['pending', 'approved', 'rejected'], default: 'pending' },
    totpSecret: String,
    totpEnabled: { type: Boolean, default: false },
    emailVerified: { type: Boolean, default: false },
    consentGiven: { type: Boolean, default: false },
  },
  { timestamps: true }
);

userSchema.methods.comparePassword = async function (password: string) {
  return bcryptjs.compare(password, this.passwordHash);
};

userSchema.pre('save', async function (next) {
  if (!this.isModified('passwordHash')) return next();
  try {
    const salt = await bcryptjs.genSalt(12);
    this.passwordHash = await bcryptjs.hash(this.passwordHash, salt);
    next();
  } catch (error) {
    next(error as any);
  }
});

export default mongoose.model<IUser>('User', userSchema);

TypeScript
// server/src/controllers/authController.ts
import { Request, Response } from 'express';
import jwt from 'jsonwebtoken';
import User from '../models/User';
import { z } from 'zod';

const registerSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
  firstName: z.string().min(2),
  lastName: z.string().min(2),
  gender: z.enum(['M', 'F']),
  dob: z.string().refine(val => new Date(val) < new Date()),
  city: z.string().min(2),
});

export const register = async (req: Request, res: Response) => {
  try {
    const validated = registerSchema.parse(req.body);

    // Check if user exists
    const existingUser = await User.findOne({ 
      $or: [{ email: validated.email }, { phone: req.body.phone }] 
    });

    if (existingUser) {
      return res.status(400).json({ error: 'User already exists' });
    }

    // Create user
    const user = new User({
      email: validated.email,
      phone: req.body.phone,
      passwordHash: validated.password,
      profile: {
        firstName: validated.firstName,
        lastName: validated.lastName,
        dob: new Date(validated.dob),
        gender: validated.gender,
        city: validated.city,
      },
      role: 'applicant',
      emailVerified: false,
      profileStatus: 'pending',
    });

    await user.save();

    // Generate verification email (implement SendGrid here)
    // await sendVerificationEmail(user.email, token);

    res.status(201).json({
      message: 'Registration successful. Check your email to verify.',
      userId: user._id,
    });
  } catch (error) {
    res.status(400).json({ error: error instanceof Error ? error.message : 'Registration failed' });
  }
};

export const login = async (req: Request, res: Response) => {
  try {
    const { email, password } = req.body;

    const user = await User.findOne({ email });
    if (!user || !(await user.comparePassword(password))) {
      return res.status(401).json({ error: 'Invalid credentials' });
    }

    if (user.status === 'suspended') {
      return res.status(403).json({ error: 'Account suspended' });
    }

    // Check 2FA for staff
    if (user.role === 'staff' && user.totpEnabled) {
      return res.status(200).json({ requiresTwoFA: true, userId: user._id });
    }

    // Generate tokens
    const accessToken = jwt.sign(
      { userId: user._id, role: user.role, email: user.email },
      process.env.JWT_SECRET!,
      { expiresIn: process.env.JWT_EXPIRE || '24h' }
    );

    const refreshToken = jwt.sign(
      { userId: user._id },
      process.env.REFRESH_TOKEN_SECRET!,
      { expiresIn: process.env.REFRESH_TOKEN_EXPIRE || '7d' }
    );

    res.json({
      accessToken,
      refreshToken,
      user: { _id: user._id, email: user.email, role: user.role, profileStatus: user.profileStatus },
    });
  } catch (error) {
    res.status(500).json({ error: 'Login failed' });
  }
};

// client/src/store/authContext.tsx
import React, { createContext, useContext, useState, useEffect } from 'react';

interface User {
  _id: string;
  email: string;
  role: 'admin' | 'staff' | 'applicant';
  profileStatus: string;
}

interface AuthContextType {
  user: User | null;
  accessToken: string | null;
  loading: boolean;
  login: (email: string, password: string) => Promise<void>;
  logout: () => void;
  register: (data: any) => Promise<void>;
}

const AuthContext = createContext<AuthContextType | null>(null);

export const AuthProvider: React.FC<{ children: React.ReactNode }> = ({ children }) => {
  const [user, setUser] = useState<User | null>(null);
  const [accessToken, setAccessToken] = useState<string | null>(null);
  const [loading, setLoading] = useState(true);

  // Check stored token on mount
  useEffect(() => {
    const token = localStorage.getItem('accessToken');
    const storedUser = localStorage.getItem('user');
    if (token && storedUser) {
      setAccessToken(token);
      setUser(JSON.parse(storedUser));
    }
    setLoading(false);
  }, []);

  const login = async (email: string, password: string) => {
    // Call API
    // setAccessToken, setUser, localStorage
  };

  const logout = () => {
    setUser(null);
    setAccessToken(null);
    localStorage.removeItem('accessToken');
    localStorage.removeItem('user');
  };

  const register = async (data: any) => {
    // Call API
  };

  return (
    <AuthContext.Provider value={{ user, accessToken, loading, login, logout, register }}>
      {children}
    </AuthContext.Provider>
  );
};
